# Memory

## User Preferences
- Prefer practical, direct help over abstract planning.
- Prefer responses that are concise but still human and natural, not robotic one-line summaries.
- Comfortable with wider reflection and conversation, but project facts should be preserved in `AGENTS.md`.
- If context gets crowded, it is acceptable to start a new conversation; keep durable project memory in repo files.

## Collaboration Style
- The user is strong in systems, storage, hardware, crypto, and recovery workflows.
- The user is less interested in spending time on Python boilerplate and repetitive scripting details; this is a good place for the assistant to help.
- The user often runs commands locally and pastes outputs back. Keep instructions executable and easy to copy.
- When changing repository memory, put project-operational facts in `AGENTS.md` and personal/collaboration memory here in `MEMORY.md`.

## User Background That Matters For Collaboration
- Long-time interest in AI, storage, electronics, firmware, and crypto.
- Has hands-on experience with RAID recovery, HBA/array hardware, SSD/NAND details, DIY electronics, and free software projects.
- Thinks in terms of storage tiers and system architecture rather than one-off tools.
- Values preserving genuinely important data, especially historical market data and crypto private keys.

## How To Be Useful
- Assume the user usually already understands the system-level problem; help most on implementation, scripting, edge cases, and making workflows resumable.
- Preserve hard-won project context in repository files so future sessions can resume quickly.
- When a task involves old legacy systems, favor solutions that reduce operational risk and simplify the final archived result.
